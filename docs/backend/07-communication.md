# 通信架构 (Communication Architecture)

> 相关文档：[架构总览](./00-overview.md) | [任务系统](./03-task-system.md) | [API 模块](./09-api.md)

## 概述

通信模块负责 Backend 与外部系统的所有数据交换，包括 PLC 控制、前端交互、视频流推送等。

**核心设计决策**：
- **双进程架构**：主进程专注实时计算，后端进程专注 API 服务
- **通信信号通道互斥**：Redis（开发调试）和 Modbus（生产环境）二选一，通过配置切换
- **抽象层解耦**：CommunicationTask 依赖 `CommSignalBase` 抽象，不依赖具体实现

---

## 1. 双进程架构

### 1.1 为什么采用双进程？

| 问题 | 双进程方案 |
|------|-----------|
| **API 崩溃** | 不影响 Vision 主循环，机械臂继续工作 |
| **内存泄漏** | API 进程可以独立重启 |
| **GIL 竞争** | 两个进程各有独立 GIL |
| **资源隔离** | Vision 独占 GPU，API 不抢资源 |
| **调试隔离** | 可以单独调试/重启 API |

### 1.2 架构总览

```
┌─────────────────────────────────────────────────────────────────────────┐
│                          双进程架构                                      │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │

│  │   主进程 (Vision)        │        │   后端进程 (FastAPI)     │        │
│  │                         │        │                         │        │
│  │  • 相机采集              │        │  • REST API             │        │
│  │  • YOLO 检测            │        │  • WebSocket            │        │
│  │  • 多帧稳定              │        │  • 数据查询              │        │
│  │  • PLC 通信 (Modbus)    │        │                         │        │
│  │                         │        │                         │        │
│  │  优先级：高              │        │  优先级：低              │        │
│  │  稳定性要求：极高         │        │  稳定性要求：一般         │        │
```

### 1.3 职责分离

| 进程 | 职责 | 稳定性要求 |
|------|------|-----------|
| **主进程** | 相机采集、YOLO 检测、多帧稳定、PLC 通信 | 极高（7×24） |
| **后端进程** | REST API、WebSocket、数据查询、前端服务 | 一般（可重启） |

### 1.4 TaskManager 主循环与通信抽象

TaskManager 通过 CommunicationTask 与外部系统通信，CommunicationTask 依赖 CommSignalBase 抽象层：

```
┌──────────────────────────────────────────────────────────────────────────────┐
│                              主进程 (Vision)                                  │
│                                                                              │
│  ┌────────────────────────────────────────────────────────────────────────┐  │
│  │                       TaskManager 主循环                                │  │
│  │                                                                        │  │
│  │  loop:                                                                │  │
│  │    # 1. 执行视觉检测                                                  │  │
│  │    result = detection_task.run_iteration(image)                      │  │
│  │                                                                        │  │
│  │    # 2. 处理通信（CommunicationTask 封装业务逻辑）                     │  │
│  │    comm_result = communication_task.run_iteration()                  │  │
│  │                                                                        │  │
│  └────────────────────────────────────────────────────────────────────────┘  │
│                                                                              │
│  ┌────────────────────────────────────────────────────────────────────────┐  │
│  │                    CommunicationTask (业务逻辑层)                       │  │
│  │                                                                        │  │
│  │  职责：                                                                │  │
│  │    • 解析消息语义（request_target / pick_done / reset）               │  │
│  │    • 调用 PickProcess 获取目标                                        │  │
│  │    • 构造响应消息                                                      │  │
│  │                                                                        │  │
│  │  依赖：CommSignalBase（注入）                                       │  │
│  └────────────────────────────────────────────────────────────────────────┘  │
│                              │                                               │
│                              │ 依赖抽象                                      │
│                              ▼                                               │
│  ┌────────────────────────────────────────────────────────────────────────┐  │
│  │                    CommSignalBase (基础设施层)                       │  │
│  │                                                                        │  │
│  │  接口（纯 I/O，不含业务语义）：                                         │  │
│  │    • receive() -> Optional[dict]   # 接收消息                          │  │
│  │    • send(message: dict) -> None   # 发送消息                          │  │
│  │    • close() -> None               # 关闭连接                          │  │
│  │                                                                        │  │
│  └────────────────────────────────────────────────────────────────────────┘  │
│                          │                         │                         │
│                          ▼                         ▼                         │
│                 ┌──────────────┐          ┌──────────────┐                   │
│                 │ModbusAdapter │          │ RedisAdapter │                   │
│                 │              │          │              │                   │
│                 │ • pymodbus   │          │ • 通信信号   │                   │
│                 │ • 配置驱动   │          │ • 命令收发   │                   │
│                 └──────────────┘          └──────────────┘                   │
│                                                                              │
│                 注意：Redis 和 Modbus 互斥，通过配置选择                      │
│                                                                              │
└──────────────────────────────────────────────────────────────────────────────┘
          │                                        │
          ▼                                        ▼
 ┌──────────────────┐                     ┌──────────────┐
 │       PLC        │                     │    Redis     │
 │  (Modbus Server) │                     │              │
 │                  │                     └──────────────┘
 │  • 触发请求      │                            │
 │  • 接收单目标    │                            ▼
 │  • 控制机械臂    │                   ┌──────────────────┐
 └──────────────────┘                   │  后端进程         │
                                        │  (FastAPI)       │
                                        │                  │
                                        │  • WebSocket     │
                                        │  • REST API      │
                                        │  • 数据查询      │
                                        └──────────────────┘
                                                │
                                                ▼
                                        ┌──────────────┐
                                        │   Frontend   │
                                        │   (Browser)  │
                                        └──────────────┘
```

**设计要点**：
- **分层清晰**：CommunicationTask 处理业务逻辑，CommSignalBase 只做 I/O
- **主进程纯粹性**：TaskManager 不直接依赖具体协议，通过抽象层解耦
- **最大隔离**：前端推送走 Redis → 后端进程 → WebSocket，不在主进程中处理 WebSocket
- **PLC 直连**：Modbus 通信在主进程内完成，保证实时性

## 2. PLC 通信 (Modbus TCP)

### 2.1 为什么选择 Modbus TCP？

| 方面 | 说明 |
|------|------|
| **协议成熟** | 工业标准，PLC 原生支持 |
| **实现简单** | pymodbus 库成熟稳定 |
| **无需中间层** | Python 直接与 PLC 通信，不需要 C Bridge |

### 2.2 通信模式：PLC 触发

采用 **PLC 触发 + Python 响应** 模式，PLC 主导节奏：

```
     PLC                                              Python
      │                                                  │
      │  1. 写入"请求标志" = 1                            │
      │ ─────────────────────────────────────────────►   │
      │                                                  │
      │                                    2. 检测到请求  │
      │                                       ↓          │
      │                                    3. 执行 Vision │
      │                                       ↓          │
      │                                    4. 选择最优目标 │
      │                                       ↓          │
      │  5. 写入结果 (x, y, status)                      │
      │ ◄─────────────────────────────────────────────   │
      │                                                  │
      │  6. 读取结果                                      │
      │                                                  │
      │  7. 清除"请求标志" = 0                            │
      │ ─────────────────────────────────────────────►   │
      │                                                  │
      │  8. 执行机械臂动作                                │
      │                                                  │
```

### 2.3 职责边界

```
┌─────────────────────────────────────────────────────────────────────┐
│                          PC 侧 (Python)                              │
│                                                                      │
│  职责：                                                              │
│    • Vision 检测、多帧稳定                                           │
│    • 目标排序和选择                                                   │
│    • 输出单个最优目标                                                 │
│                                                                      │
│  不关心：寄存器地址、PLC 内部逻辑、机械臂控制                          │
└─────────────────────────────────────────────────────────────────────┘
                                │
                                │ Modbus TCP (抽象接口)
                                ▼
┌─────────────────────────────────────────────────────────────────────┐
│                          PLC 侧                                      │
│                                                                      │
│  职责：                                                              │
│    • 触发定位请求                                                    │
│    • 接收单个目标坐标                                                 │
│    • 控制机械臂                                                      │
│    • 定义寄存器映射                                                   │
│                                                                      │
│  不关心：Vision 算法、目标排序逻辑、多目标管理                         │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.4 设计原则

- **单目标输出**：每次请求只返回一个最优目标，复杂性留在 PC 侧
- **数据映射抽象**：寄存器地址由 PLC 定义，PC 侧通过配置读取，不硬编码
- **同步设计**：使用 pymodbus 同步客户端，简单可靠

## 3. 通信通道

系统有两个独立的通信通道，各司其职：

```
┌─────────────────────────────────────────────────────────────────┐
│  通信通道                                                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  【通信信号通道】互斥，通过 CommSignalBase                        │
│    • 业务语义：request_target / pick_done / reset               │
│    • Redis 模式（开发调试）OR Modbus 模式（生产）                │
│    • CommunicationTask 解析业务语义                             │
│                                                                  │
│  【事件推送通道】始终 Redis Stream                               │
│    • 检测事件、状态更新、Pick 结果                               │
│    • 主进程 → Redis Stream → 后端进程 → WebSocket → 前端        │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 3.1 通信信号通道

通信信号通道用于挑取业务的核心交互，通过 CommSignalBase 抽象，支持两种互斥的实现：

| 模式 | 实现 | 用途 | 场景 |
|------|------|------|------|
| **Redis** | RedisAdapter | 开发调试、前端验证 | 无机械臂环境，通过前端 API 模拟 PLC 行为 |
| **Modbus** | ModbusAdapter | 生产环境 | PLC 直连，机械臂控制 |

通过配置文件选择：

```yaml
communication:
  comm_signal:
    type: redis  # 或 modbus
```


**业务语义**（由 CommunicationTask 解析）：
- `request_target`：请求下一个挑取目标
- `pick_done`：通知挑取完成
- `reset`：重置业务状态（不停止主循环、不做系统级重启）

**Redis 模式建议约定**：
- 通信信号队列使用独立 key（例如 `pluck:control`），避免与其他 Redis 列表混用

### 3.2 事件推送通道

事件推送通道**始终使用 Redis Stream**，不受通信信号通道模式影响：

```
主进程 → Redis Stream (pluck:events) → 后端进程 → WebSocket → 前端
```

**推送内容**：
- 检测事件、状态更新
- 帧计数、FPS、运行状态
- Pick 结果通知

## 4. 通信抽象层

通信抽象层提供统一的消息收发接口，使 TaskManager 和 CommunicationTask 不依赖具体的通信协议实现。

**核心原则**：
- **抽象层只做 I/O**：`receive()` / `send()` 纯消息收发，不含业务逻辑
- **领域逻辑在 Task 层**：`request_target`、`pick_done` 等业务语义由 CommunicationTask 处理
- **实现互斥**：Redis 和 Modbus 二选一，通过配置切换

### 4.1 架构分层

```
┌─────────────────────────────────────────────────────────────────────────┐
│                           架构分层                                       │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                          │

│                                                                          │
│    ┌──────────────────────────────────────────────────────────────┐     │
│    │                    CommunicationTask                          │     │
│    │                                                               │     │
│    │  职责：                                                       │     │
│    │    • 解析消息语义（request_target / pick_done / reset）       │     │
│    │    • 调用 PickProcess 获取目标                                │     │
│    │    • 构造响应消息                                             │     │
│    │                                                               │     │
│    │  依赖：CommSignalBase（注入）                              │     │
│    └──────────────────────────────────────────────────────────────┘     │
│                              │                                           │
│                              │ 依赖抽象                                  │
│                              ▼                                           │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│  【基础设施层】通信抽象                                                  │
│                                                                          │
│    ┌──────────────────────────────────────────────────────────────┐     │
│    │                   CommSignalBase                           │     │
│    │                                                               │     │
│    │  接口（纯 I/O，不含业务语义）：                                │     │
│    │    • receive() -> Optional[dict]   # 接收消息                 │     │
│    │    • send(message: dict) -> None   # 发送消息                 │     │
│    │    • close() -> None               # 关闭连接                 │     │
│    └──────────────────────────────────────────────────────────────┘     │
│                              │                                           │
│                              │ 实现                                      │
│                              ▼                                           │
│    ┌────────────────────┐         ┌────────────────────┐                │
│    │  RedisAdapter      │   OR    │  ModbusAdapter     │                │
│    │                    │         │                    │                │
│    │  • 开发/调试用     │         │  • 生产环境        │                │
│    │  • 前端验证        │         │  • PLC 通信        │                │
│    └────────────────────┘         └────────────────────┘                │
│                                                                          │
│    注意：Redis 和 Modbus 互斥，通过配置选择                              │
│                                                                          │
└─────────────────────────────────────────────────────────────────────────┘
```

## 5. 前端通信

### 5.1 WebSocket

向前端推送实时状态更新（帧数、检测数、FPS 等）。

**数据流**：
```
主进程 → Redis Stream → 后端进程 → WebSocket → 前端
```

**消息内容**：
- 帧计数、检测数、稳定目标数
- FPS、运行状态
- Pick 结果通知

### 5.2 MJPEG 视频流

向前端推送实时视频流（带标注的检测结果）。

**数据流**：
```
主进程 → FramePublisher → 后端进程 → HTTP StreamingResponse → 前端
```

**特点**：
- JPEG 编码，质量可配置
- 帧率限制，避免带宽浪费
- 标注框显示稳定目标

详见：[API 模块](./09-api.md)

## 6. 架构要点总结

| 要点 | 说明 |
|------|------|
| **双进程隔离** | 主进程专注计算，后端进程专注服务，互不影响 |
| **通信信号通道互斥** | Redis（开发调试）和 Modbus（生产）二选一 |
| **事件通道独立** | 事件推送始终走 Redis，不受通信信号模式影响 |
| **PLC 直连** | Modbus TCP 直接通信，无需中间层 |
| **PLC 触发模式** | PLC 主导节奏，Python 响应请求 |
| **单目标输出** | 复杂性留在 PC 侧，PLC 只接收一个目标 |
| **抽象层解耦** | TaskManager 通过 CommunicationTask 间接依赖 CommSignalBase |
| **数据映射抽象** | 寄存器地址由 PLC 定义，PC 侧配置驱动 |
